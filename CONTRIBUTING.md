# CONTRIBUTING.md — правила разработки Hookline

Закон проекта. Не согласен — PR в этот файл и убеди лида. Пока не смержено — действует как написано.

---

## 1. Быстрый старт (онбординг за 30 минут)

```bash
git clone https://github.com/<org>/hookline && cd hookline
cp .env.example .env
make up              # compose: postgres, api, 3 воркера, sink, grafana
make migrate
make demo            # создаёт app + endpoint на sink + подписку и шлёт тестовое событие
make test            # go test -race ./...
make lint
```

Требования: Go 1.26+, Docker, make. Не завелось за 30 минут — заведи issue с меткой `onboarding`: это баг документации, а не твой.

Первая задача новичка — L0 из `docs/STARTER_TASKS.md`, полный цикл до мержа за 3 дня.

## 2. Задачи и доска (GitHub Projects)

- Единственный трекер — **GitHub Issues + доска GitHub Projects** этого репозитория. Задач вне трекера не существует. Как пользоваться доской и скрам-минимум — `docs/GITHUB_PROJECTS_GUIDE.md`, читается один раз до первого спринта.
- Колонки (поле `Status`): `Backlog → Ready → In Progress → In Review → Done`.
- Поля задачи: **Level** (L0–L4), **Size** (S/M/L), **Sprint** (итерация, недельная), **Module** (queue, worker, ingest, delivery, signing, backoff, matcher, breaker, attempts, api, web, infra, docs). Плюс метки `good-first-issue`, `blocked`, `flaky`.
- Берём только из `Ready`; назначил себя (Assignee) → перетащил в `In Progress`. Максимум **2 задачи In Progress** на человека.
- Лестница уровней: до L3 — сначала ≥2 задачи L1 и ≥1 L2; первая L3 только в паре с лидом или стажёром.
- **Связь issue ↔ PR:** в описании PR обязательна строка `Closes #123`. Тогда при мерже issue закрывается сам и уезжает в `Done` — вручную ничего двигать не надо. В `In Review` карточка тоже переезжает автоматически при открытии PR.
- Ветка называется по номеру issue: `feat/123-claim-query`. Номер — единственный «клей» между кодом и задачей.
- Завис > 2 дней → метка `blocked` на issue, комментарий с описанием затыка и сообщение в `#blockers`. Обязательно.

## 3. Ветки и коммиты

- `main` защищена: прямой пуш и force-push запрещены всем, включая лида; мерж только через PR с зелёным CI и аппрувом.
- Ветка: `feat/123-claim-query`, `fix/207-lease-expiry` (число = номер issue). Живёт ≤ 5 дней.
- Conventional Commits: `feat(queue): add skip locked claim`, `fix(worker): return lease on shutdown`. Проверяется CI.
- Мерж — squash; перед ревью `git fetch && git rebase origin/main`.

## 4. Pull Request

- **≤ 300 изменённых строк.** Больше — дели; это защита ревьюера, а не наказание.
- Шаблон: что сделано / как проверить / вывод команды или скрин / обязательная строка `Closes #123`.
- Без тестов на новую бизнес-логику PR не ревьюится. Draft PR с первого дня приветствуется.
- **SLA ревью 24 ч** (выходные 48). Модули `queue/`, `worker/`, `signing/` — 2 аппрува, один из них лид.
- На каждый комментарий ревью — ответ словами (исправил / возражаю потому-то / вынес в отдельный issue). Молчаливый resolve запрещён.
- Как ревьюить: пойми задачу → запусти ветку → «предлагаю X, потому что Y» → различай `blocking:` и `nit:` → отмечай хорошее.

## 5. Стандарты кода

Общие (как в любом нашем проекте):
- `golangci-lint fmt` (gofumpt) + линтер v2, пресет standard + `errcheck, staticcheck, revive, gosec, errorlint, sloglint, testifylint`. Отключение правил — только PR-ом в конфиг.
- Ошибки: `fmt.Errorf("claim: %w", err)`, `errors.Is/As`, доменные ошибки в `internal/domain/errors.go`, `panic` запрещён.
- `ctx context.Context` первым аргументом везде, где I/O; в структурах не хранить.
- Интерфейсы объявляет потребитель; зависимости — через конструкторы в main.go; глобального состояния и `init()` с логикой нет.
- Логи только `slog` c request_id / message_id / worker_id.

Специфичные для Hookline (нарушение = блокер на ревью):
1. **`time.Now()` в бизнес-логике запрещён.** Время берётся из интерфейса `Clock` (ADR-0004) — иначе тесты ретраев и лизов невозможно писать без `time.Sleep`.
2. **`time.Sleep` в тестах запрещён.** Фейковые часы + `synctest`/каналы. Тест, который «иногда падает», — это баг теста, а не «флак».
3. **Статусы `messages` меняются только через пакет `queue`.** Любой `UPDATE messages SET status` вне `queue` — блокер.
4. **Любой claim-запрос содержит `FOR UPDATE SKIP LOCKED`.** Без него при N воркерах будет двойная доставка; ADR-0003 — обязателен к прочтению до правок очереди.
5. **Секреты эндпоинтов не логируются и не возвращаются API после создания.** Заголовки в `attempts` пишутся с редактированием чувствительных значений.
6. **Каждая горутина имеет владельца и путь завершения** (context/канал). «go func и забыл» — блокер. Воркеры и reaper останавливаются по SIGTERM в тестируемом порядке.
7. **Сравнение подписей — только `hmac.Equal`** (constant-time). `==` на HMAC — блокер и повод для отдельного разговора о тайминг-атаках.
8. Доставка наружу — только через `internal/delivery` (там таймауты и лимиты). Голый `http.Get` в бизнес-коде запрещён.

## 6. Тесты

- Table-driven по умолчанию; `make test` = `go test -race ./...` всегда.
- Юнит — без сети и БД. Интеграционные (`//go:build integration`) — testcontainers + `cmd/sink` как управляемый получатель, `make test-integration`.
- 100% покрытие: `signing`, `backoff`, `matcher`; тестовые векторы подписи зафиксированы навсегда (только добавлять).
- **Обязательные интеграционные тесты ядра** (без них PR в queue/worker не мержится):
  1. 2+ воркера не берут одно сообщение (`SKIP LOCKED` работает);
  2. воркер «умер» посреди доставки → сообщение подхватывается после истечения лиза;
  3. N сообщений при 3 воркерах доставлены ровно по разу (при исправном получателе);
  4. неуспех → перенос по backoff, исчерпание попыток → `dead`;
  5. повтор `Idempotency-Key` не создаёт второе событие.
- Нашёл баг → сначала красный тест, потом фикс.

## 7. Коммуникация и ритуалы

- Discord: `#general`, `#dev`, `#blockers`, `#review`, `#demo`.
- Пн 30–45 мин планирование (голос). Ср — асинхронный стендап в `#dev` (3 строки). Пт 30 мин — демо смерженного. Раз в месяц — ретро.
- Решение, которого нет в issue / ADR, не существует. ADR: `docs/adr/`, полстраницы, PR-ом.
- Новичкам: `docs/GIT_FOR_BEGINNERS.md` + `docs/STARTER_TASKS.md`.
- **Ритуал «объясни у доски»:** на каждом втором демо один участник за 5 минут объясняет команде одну строку из таблицы §9 ТЗ (backoff, SKIP LOCKED, at-least-once…). Это тренировка ответа на собеседовании, и она обязательна для всех по очереди.

## 8. Участие и отвал

- Загрузка 6–8 ч/нед. Пропадание на 2 недели без предупреждения → задачи и модуль переходят другому; вернуться можно всегда через задачу из Ready.
- Конфликты по коду: аргументы + ADR; финальное слово у лида с письменным обоснованием.

## 9. Стоп-лист скоупа

- **Kafka, RabbitMQ, NATS, Redis-очереди — запрещены до v1.0.** Вся суть проекта в том, что очередь на Postgres мы пишем и понимаем сами.
- Kubernetes, микросервисное дробление, мультитенантность, биллинг, UI-красота — в backlog «после v1.0».
- Смена стека — только ADR, дефолтный ответ «нет».
