# GitHub Projects + Scrum: как мы работаем в Hookline

Прочитать один раз, до первого спринта. Прочитал → ✅ в Discord `#general`. На созвонах это больше не объясняем.

Почему GitHub Projects, а не Weeek/Jira: бесплатный трекер без лимита на людей, живёт прямо в репозитории, и — главное — issue, ветка и pull request связаны автоматически. Одна строка в описании PR закрывает задачу и двигает карточку на доске. Меньше ручной возни, меньше «забыл передвинуть».

---

## Часть 1. Что где лежит

- **Issue** — сама задача (описание, критерии приёмки, обсуждение). Живёт в репозитории, вкладка Issues.
- **Project (доска)** — вид на эти issue: колонки, поля, фильтры. Вкладка Projects. Одна доска на весь проект.
- **Метки (labels)** — быстрые пометки: `good-first-issue`, `blocked`, `flaky`, `bug`, `docs`.
- **Поля доски** (видны только на доске, не в самом issue):
  - `Status`: Backlog → Ready → In Progress → In Review → Done
  - `Level`: L0, L1, L2, L3, L4
  - `Size`: S (≤2ч), M (до дня), L (2–3 дня)
  - `Sprint`: недельная итерация (S1…S8)
  - `Module`: queue, worker, ingest, delivery, signing, backoff, matcher, breaker, attempts, api, web, infra, docs

Настройку всего этого делает лид один раз по `docs/REPO_SETUP.md`.

## Часть 2. Жизненный цикл задачи

```
Лид создаёт issue #123 ─────────────────────────────→ Status: Backlog
Задача дозрела (описание + критерии + Level/Size) ──→ Status: Ready
Ты берёшь: Assignee = ты, тащишь карточку ─────────→ Status: In Progress
Ветка feat/123-claim-query, пишешь код
Открыл PR c "Closes #123" в описании ──────────────→ Status: In Review  (автоматически)
PR смержен ────────────────────────────────────────→ issue закрыт, Status: Done (автоматически)
```

Что делаешь руками: назначаешь себя, тащишь в `In Progress`, пишешь `Closes #123` в PR. Дальше доска двигается сама.

**Правила:**
- Берём задачи **только из `Ready`**. Если в задаче нет критериев приёмки — не бери, напиши комментарий лиду, пусть дозреет.
- Максимум **2 задачи In Progress** на человека. Хочешь взять третью — сначала домержи одну.
- Всё обсуждение задачи — **в комментариях issue**, а не в личке и не в голосовом. Договорились голосом → одна строка в issue. Issue — это память проекта; через месяц никто не вспомнит, почему сделали именно так.
- Застрял → метка `blocked` + комментарий «жду X / не понимаю Y» + сообщение в `#blockers`.
- Нашёл баг — заводишь issue сам, шаблон подставится автоматически. Для багов доставки обязательно `message_id` и вывод `GET /messages/{id}`, иначе баг невоспроизводим.

**Шаблон описания задачи** (подставляется автоматически при создании issue):
```
Что: одним предложением.
Зачем: § из TZ.md.
Модуль: queue / signing / ...
Критерии приёмки:
  - [ ] проверяемое условие
Где в коде: пакеты/файлы.
Подсказки: ссылки (для L0–L1 обязательны).
```

## Часть 3. Скрам-минимум (глоссарий, чтобы не путаться в терминах)

Мы используем не канонический Scrum, а его лёгкую версию. Вот все слова, которые будут звучать на созвонах — эти же термины используются в Jira и на работе, поэтому учим сразу «взрослые» названия.

| Термин | Что это у нас |
|---|---|
| **Спринт (итерация)** | Неделя работы (пн–вс) с фиксированным набором задач, всего S1–S8. Внутрь спринта задачи не докидываются: новое падает в Backlog и ждёт следующего планирования. |
| **Бэклог** | Колонка Backlog — всё, что когда-нибудь надо сделать, отсортировано по приоритету сверху вниз. Приоритеты расставляет лид. |
| **Планирование** | Пн, 30–45 мин, голосом: смотрим остатки прошлой недели → лид проставляет задачам Sprint → каждый называет, что берёт. Выходим с личным планом на неделю. |
| **Стендап** | Ср, асинхронно, текстом в `#dev`: сделал / делаю / блокеры — 3 строки. Это не отчёт начальнику, а способ поймать блокер до пятницы. |
| **Демо (обзор спринта)** | Пт, 30 мин: показываем работающее руками, с экрана. Показывается **только смерженное в main**; «почти готово» не демонстрируется. Каждое второе демо — плюс 5-минутный «разбор у доски» (CONTRIBUTING §7). |
| **Ретро** | Раз в месяц, 30–40 мин: что помогало / что мешало / 1–2 конкретных изменения в процессе. Про процесс, не про виноватых. |
| **Критерии приёмки** | Чек-лист внутри issue. Спор «готово / не готово» разрешается только тыканием в эти пункты. |
| **DoD (Definition of Done)** | Общий чек-лист качества поверх критериев приёмки: тесты, линтер, ревью, зелёный CI (TZ §7). Критерии приёмки — про конкретную задачу, DoD — про любую. |
| **Оценка (Size)** | S/M/L вместо стори-поинтов. Ставит исполнитель при взятии. Ошибся — нормально, скажи на стендапе. Задач больше L не существует: режем на части. |
| **Velocity** | Сколько Size команда реально закрывает за спринт. Смотрим после S2–S3 и планируем не больше этого. Это не соревнование и не оценка людей. |
| **Блокер** | То, из-за чего ты не можешь продолжать. Правило: 30 минут гуглишь сам → 2 часа мучаешься → пишешь в `#blockers`. Молчать о блокере — единственная настоящая ошибка в этом проекте. |
| **Груминг** | Лид (иногда со стажёром) раз в неделю доводит верх Backlog до состояния Ready. Остальным приходить не надо. |
| **Флак (flaky-тест)** | Тест, который иногда падает. У нас почти всегда означает `time.Sleep` или реальные часы вместо интерфейса `Clock`. Флак = issue с меткой `flaky`, чинится в тот же спринт. Перезапускать CI «пока не позеленеет» — запрещено. |

Чего из настоящего Scrum у нас нет и почему: скрам-мастера (роль на лиде), покера планирования (S/M/L на глаз достаточно), burndown-графиков (пять человек видят доску целиком), ежедневных дейликов (у всех учёба/работа, хватает среды).

## Часть 4. Календарь недели (повторяется 8 раз)

```
Пн  19:00–19:45  Планирование (голос, обязательно)
Ср  до 21:00     Асинхронный стендап в #dev (текст, 3 строки)
Пт  19:00–19:30  Демо (+ раз в 2 недели «разбор у доски»)
Вс               Дедлайн PR, чтобы попасть в демо следующей недели
Раз в месяц      Ретро вместо части демо
```

Пропускаешь созвон — предупреди заранее и оставь письменный апдейт в `#dev`. Два молчаливых пропуска подряд — разговор с лидом о загрузке (без драмы: может, просто пора снизить объём).

## Часть 5. Мини-шпаргалка по интерфейсу

- **Взять задачу:** Issues → открыть → справа `Assignees` → себя → на доске перетащить в `In Progress`.
- **Найти, что взять:** на доске сохранённый вью `📥 Ready` (уже отфильтровано) или фильтр `label:good-first-issue`.
- **Связать PR с задачей:** в описании PR строка `Closes #123` (можно `Fixes #123`). Проверь, что в правой панели PR появилась ссылка на issue — значит связь есть.
- **Посмотреть свои задачи:** фильтр `assignee:@me` или сохранённый вью `🙋 Мои`.
- **Что горит:** вью `🔥 Blocked` (метка `blocked`) — на планировании смотрим его первым.
