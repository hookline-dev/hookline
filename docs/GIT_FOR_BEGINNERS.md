# Git и GitHub с нуля: полная инструкция для новичков

Этот файл — для тех, кто **впервые** работает с git и GitHub. Читай сверху вниз, не пропуская. Держи его открытым в соседней вкладке первые пару недель — сюда можно возвращаться.

Главное, что нужно понять сразу и запомнить навсегда: **в этом проекте ты физически не можешь ничего сломать.** Главная ветка (`main`) защищена настройками репозитория — напрямую в неё не может запушить никто, даже тимлид. Твой код попадает туда только после того, как робот проверит его тестами, а живой человек — прочитает. Худшее, что реально может случиться, — беспорядок в *твоей личной* ветке, а её всегда можно удалить и начать заново. Поэтому не бойся жать кнопки и пробовать.

---

## Часть 0. Что это вообще такое (2 минуты теории)

- **Git** — программа на твоём компьютере, которая запоминает историю изменений в коде. Как «сохранения» в игре: можно вернуться к любому.
- **GitHub** — сайт, где эти истории хранятся в облаке, чтобы вся команда работала над одним проектом.
- **Репозиторий (repo)** — папка проекта со всей историей. Наш: `github.com/hookline-dev/hookline`.
- **Коммит (commit)** — одно «сохранение»: снимок твоих изменений с подписью, что ты сделал.
- **Ветка (branch)** — отдельная линия работы. Ты делаешь свою задачу в своей ветке, чтобы не мешать другим. Главная ветка называется `main`.
- **PR (pull request)** — заявка «влейте мои изменения из моей ветки в main». Её смотрят, обсуждают и одобряют.

Рабочая картинка целиком: *взял задачу → создал свою ветку → написал код → сохранил (коммит) → отправил на GitHub (пуш) → открыл PR → тебя проверили → влили в main.*

## Часть 1. Разовая настройка (один вечер, по шагам)

### 1.1. Аккаунт на GitHub
Зарегистрируйся на github.com. Ник выбирай приличный — этот профиль потом пойдёт в резюме, работодатели его смотрят.

### 1.2. Включи двухфакторную аутентификацию (2FA)
Settings → Password and authentication → Two-factor authentication → включи через приложение (Google Authenticator, Authy и т.п.).
**Без 2FA в проект не добавят** — это защита от угона аккаунта, у нас это обязательно для всех.

### 1.3. Установи git
- Скачай с git-scm.com.
- **Windows:** вместе с git ставится программа **Git Bash** — работай в ней, а не в стандартной командной строке (в Git Bash работают все команды из этого файла).
- **macOS:** открой Terminal, набери `git --version` — если git не стоит, система сама предложит установить.

### 1.4. Представься git-у
Git подписывает каждый твой коммит именем и почтой. Задай их один раз (вставляй по одной строке, после каждой — Enter):

```bash
git config --global user.name "Имя Фамилия"
git config --global user.email "почта-с-которой-регался-на-github"
git config --global pull.rebase true
git config --global init.defaultBranch main
```

Почта должна совпадать с той, что на GitHub, — тогда коммиты привяжутся к твоему профилю.

### 1.5. Настрой вход по SSH-ключу (чтобы не вводить пароль каждый раз)

SSH-ключ — это пара файлов: секретный (остаётся у тебя) и публичный (отдаёшь GitHub). Создаём:

```bash
ssh-keygen -t ed25519 -C "твоя почта"
```
Три раза нажми Enter (соглашаешься с местом хранения и пустым паролем). Теперь покажи публичную часть:

```bash
cat ~/.ssh/id_ed25519.pub
```
Скопируй **весь вывод** (начинается с `ssh-ed25519` и заканчивается твоей почтой). На GitHub: Settings → **SSH and GPG keys** → **New SSH key** → вставь в поле Key → Add.

Проверь, что связь работает:
```bash
ssh -T git@github.com
```
Если увидишь `Hi <ник>! You've successfully authenticated` — готово. (На первый вопрос `Are you sure...?` ответь `yes`.)

### 1.6. Установи редактор кода
**VS Code** (code.visualstudio.com) + расширение **Go** (при первом открытии `.go` файла оно само предложит доставить инструменты — соглашайся). Либо GoLand, если есть студенческая/бесплатная лицензия.

### 1.7. Скачай проект себе (клонирование)
Перейди в терминале в папку, где держишь проекты, и:
```bash
git clone git@github.com:hookline-dev/hookline.git
cd hookline
```
`clone` скачивает весь проект с историей. Делается один раз. Дальше просто работаешь внутри папки `hookline`.

После этого пройди `docs/onboarding.md`, чтобы поднять проект локально. ⚠️ Обязательно прочитай плашку «текущий статус проекта» в начале того файла: на ранних неделях часть команд ещё не существует, и это нормально.

## Часть 2. Единственный рабочий цикл, который тебе нужен

95% твоей работы с git — это восемь команд ниже. Выучи их порядок — остальное придёт.

```bash
# ── ШАГ 0. Начало новой задачи: обнови main и создай от него свою ветку ──
git switch main            # перешли на главную ветку
git pull                   # скачали свежие изменения команды
git switch -c feat/123-korotkoe-opisanie   # создали свою ветку и перешли в неё
#            ↑ feat/ + номер задачи из GitHub + пара слов о сути. Пример: feat/45-backoff-calculator

# ── ШАГ 1. Поработал в коде. Смотрим, что изменилось ──
git status                 # какие файлы тронуты (красное = не сохранено)
git diff                   # что именно изменилось построчно (q — выйти из просмотра)

# ── ШАГ 2. Сохраняем изменения (коммит) ──
git add .                  # взять все изменения... (или "git add путь/к/файлу" — только конкретный)
git commit -m "feat: add backoff calculator with jitter"
#                ↑ формат сообщения важен, см. часть 4

# ── ШАГ 3. Отправляем ветку на GitHub (пуш) ──
git push -u origin feat/123-korotkoe-opisanie   # -u только в ПЕРВЫЙ push этой ветки
#  дальше в этой же ветке достаточно просто:  git push
```

После первого `push` терминал выдаст ссылку вида `https://github.com/.../pull/new/feat/123-...`. Открой её → заполни шаблон PR → **Create pull request**.

Дальше PR живёт сам. Пришли замечания на ревью? Правишь код в **той же ветке** → снова `git add .` + `git commit -m "..."` + `git push`. PR обновится автоматически, пересоздавать ничего не надо.

> Совет: ветку можно создать прямо на сайте — в issue справа есть кнопка **Create a branch**. Она сама даст правильное имя. Тогда локально останется только `git switch main && git pull && git switch <имя-ветки>`.

## Часть 3. Что настроено, чтобы ты не мог навредить

| Страховка | Что делает |
|---|---|
| Защита `main` | Прямой пуш в главную ветку запрещён всем. Удалить или «переписать» её нельзя. Только через PR. |
| Проверки CI | На каждый PR робот сам собирает проект, гоняет тесты и линтер. Красный CI = влить нельзя. Это не «ты плохой» — это страховка сделала свою работу. |
| Сканер секретов | Ищет случайно попавшие пароли/токены и роняет проверку, если нашёл. |
| `.gitignore` | Файл со списком того, что git игнорирует (твой локальный `.env`, временные файлы). Их нельзя закоммитить случайно. |
| Squash merge | Все твои мелкие коммиты («fix», «еще раз», «ну работай же») при вливании схлопнутся в один аккуратный. Поэтому коммить смело и часто. |
| Обязательное ревью | Минимум один человек прочитает код до вливания. Вопросы на ревью — это обучение, а не экзамен. |

### Правила личной безопасности (соблюдать всегда)
- **Никогда не коммить секреты:** пароли, токены, приватные ключи, свой `.env`. Сомневаешься — спроси в `#dev` до коммита. Если уже закоммитил секрет — **не** «удаляй следующим коммитом» (в истории он останется!), сразу напиши тимлиду: сменим/отзовём и почистим вместе.
- **`git push --force` — запрещён.** Вообще. Кажется, что нужен, — зови тимлида.
- **Не выполняй команды из интернета/от нейросетей, если не понимаешь каждое слово** — особенно `git reset --hard`, `git clean`, `rm -rf`. Спросить в `#blockers` быстрее, чем восстанавливать день работы.
- Один PR = одна задача. Не трогай «заодно» чужие файлы.
- Перед `git add .` всегда сначала `git status` — глазами проверь, что добавляешь именно то, что хотел.

## Часть 4. Как называть коммиты (Conventional Commits)

У нас правило: сообщение коммита начинается с типа изменения. Это не каприз — по этим префиксам потом автоматически собирается список изменений релиза, и робот в CI проверяет формат.

```
feat: короткое описание      — новая функциональность
fix: короткое описание       — исправление бага
docs: короткое описание      — только документация
test: короткое описание      — только тесты
refactor: короткое описание  — переписал код, поведение не менялось
chore: короткое описание     — рутина (конфиги, зависимости, CI)
```

Хорошо: `feat: add HMAC signature verification`
Плохо: `v1.0.3`, `фиксы`, `работа`, `asdf`, `Update main.go`

Описание — в повелительном наклонении и по сути («add», «fix»), маленькой буквой, без точки в конце. На русском тоже можно, главное — префикс и суть: `fix: чиню падение парсера на пустом ответе`.

## Часть 5. «Я всё сломал» — аптечка (сначала команда, потом паника)

| Ситуация | Что делать |
|---|---|
| Хочу отменить правки в файле (ещё не коммитил) | `git restore путь/к/файлу` |
| Добавил в `git add` лишний файл | `git restore --staged путь/к/файлу` |
| Ошибся в тексте последнего коммита (ещё не пушил) | `git commit --amend -m "правильный текст"` |
| Запутался в ветке / начал не от main | Не чини. Сохрани изменённые файлы куда-нибудь в сторонку, `git switch main && git pull`, создай ветку заново, верни файлы. 10 минут — и чистая жизнь. |
| `git pull` ругается на конфликт | Открой файлы с метками `<<<<<<<`, `=======`, `>>>>>>>`; оставь правильный вариант, удали метки; `git add` + `git commit`. Непонятно — скинь скрин в `#blockers`, разберём за 5 минут. |
| Случайно закоммитил `.env` или секрет | НЕ пушь. Уже запушил — сразу тимлиду, молча не чини. |
| Кажется, потерял коммиты | Ничего не трогай, зови тимлида: `git reflog` почти всегда всё возвращает. Закоммиченное в git потерять очень трудно. |
| `git push` отклонён с "repository rule violations" | Ты пытаешься запушить в `main` напрямую — так нельзя. Работай через свою ветку и PR. |

## Часть 6. Твой первый PR — чек-лист

- [ ] Ветка называется `feat/<номер>-...` и создана от свежего `main`
- [ ] Локально `make lint` и `make test` — зелёные
      _(если `make` ещё не настроен в проекте — используй `go test -race ./...` и `go vet ./...`, см. плашку в начале `docs/onboarding.md`)_
- [ ] `git status` чистый, лишних файлов не добавлено
- [ ] PR не гигантский (у нас правило ≤ 300 изменённых строк — если больше, дели задачу)
- [ ] Заполнен шаблон PR, в описании есть `Closes #<номер задачи>`
- [ ] На комментарии ревьюера отвечаешь словами (исправил / не согласен потому-то), а не молча

Прочитал файл до конца? Напиши в `#general`: **«git-гайд ✅»** — и бери первую задачу с меткой `good-first-issue` из колонки Ready на доске проекта.

Застрял на чём угодно из этого файла дольше 30 минут — это не стыдно, это ожидаемо. Пиши в `#blockers`, для того он и есть.
