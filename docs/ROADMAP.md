# 1. Кто есть кто

В таблицах задач исполнители обозначены коротко. Вот расшифровка — впиши сюда реальные имена, когда роли распределятся:

| Обозначение | Роль | Что пишет | Кто это у нас |
|---|---|---|---|
| **Лид** | тимлид, архитектор | очередь (`queue`), воркеры (`worker`), инфраструктура, CI | _вписать имя_ |
| **Ст** | бэкендер-стажёр | приём событий (`ingest`), доставка (`delivery`), предохранитель (`breaker`) | _вписать имя_ |
| **A** | разработчик A | «чистые» модули: подпись (`signing`), задержки ретраев (`backoff`), матчинг подписок (`matcher`) | _вписать имя_ |
| **B** | разработчик B | HTTP-слой (`api`), веб-дашборд | _вписать имя_ |
| **C** | разработчик C | журнал попыток (`attempts`), тестовый приёмник (`sink`), метрики, документация | _вписать имя_ |

**Что значит «владеет модулем»:** ты главный по этому куску кода. Тебя автоматически зовут ревьюером, когда кто-то его правит, и с тебя спрашивают, если он сломался. Это **не** значит, что другим туда нельзя — просто согласуй.

🟩 **Хорошая новость для новичков (A, B, C):** ваши модули устроены так, что **не зависят от ядра**. Пока тимлид пишет очередь, вы уже пишете свой код. Никто никого не ждёт.

# 2. Что значат уровни задач L0–L4

У каждой задачи на доске есть поле `Level`. Это не оценка тебя — это подсказка, что задача потребует.

| Уровень | Что это | Пример | Нужно уметь |
|---|---|---|---|
| **L0** | правки текста и конфигов, ничего не сломать невозможно | добавить себя в `team.md`, описать переменные в `.env.example` | только git |
| **L1** | простая функция или тест, без базы и сети | посчитать задержку ретрая, проверить совпадение строк | базовый Go, функции, тесты |
| **L2** | код с базой или HTTP, но однопоточный | сохранить событие в БД, написать хендлер | Go + SQL или `net/http` |
| **L3** | **конкурентность**: горутины, каналы, `context`, гонки | пул воркеров, фоновый reaper, graceful shutdown | горутины, каналы, `select` |
| **L4** | архитектурно сложное, обычно у лида | дизайн очереди целиком | опыт |

**Правило лестницы:** прежде чем брать первую L3, надо закрыть минимум 2 задачи L1 и 1 задачу L2. И первая L3 — **обязательно в паре** с тимлидом или стажёром (созвон с шарингом экрана, час-полтора). Это не формальность: конкурентность — тот раздел, где в одиночку легко написать код, который «вроде работает», а падает раз в неделю в проде.

К концу проекта **каждый** должен закрыть хотя бы одну L3. Это одна из главных целей всей затеи.

# 3. Словарь слов из этого файла

### Процессные слова

| Слово | Что значит |
|---|---|
| **Фаза** | крупный этап проекта (0–4), 1–2 недели. В GitHub оформляется как **Milestone** |
| **Спринт (S1…S8)** | одна рабочая неделя с зафиксированным набором задач. В GitHub — поле `Sprint` (тип Iteration) |
| **Milestone** («веха») | группировка задач по фазе. Показывает прогресс-бар «закрыто 12 из 18» |
| **Issue** | задача на GitHub. Одна строка в таблицах ниже = 1–3 issue |
| **Груминг** | лид раз в неделю дописывает у задач критерии приёмки и переводит их в колонку `Ready`. Остальным приходить не надо |
| **Exit-критерий** | проверяемое условие «фаза сдана». Не «вроде готово», а конкретное действие, которое можно показать |
| **Вью (view)** | сохранённая вкладка на доске с фильтром: «мои задачи», «текущий спринт», «застрявшие» |
| **Assignee** | исполнитель задачи. Ставишь себя сам, когда берёшь задачу |
| **Velocity** | сколько задач команда реально закрывает за неделю. Нужно, чтобы не планировать больше, чем успеваем. Не соревнование |
| **Скоуп** | объём работ. «Срезать скоуп» = отказаться от части фич, чтобы успеть в срок |
| **Флак (flaky-тест)** | тест, который иногда падает без причины. У нас почти всегда означает спрятанную гонку или `time.Sleep` в тесте |
| **ADR** | короткая записка «мы решили делать так, потому что…». Полстраницы, лежит в `docs/adr/` |
| **DoD** | Definition of Done — общий чек-лист «задача готова»: тесты, линтер, ревью, зелёный CI |
| **E2E-проверка** | end-to-end, «из конца в конец»: прогнать весь сценарий целиком, как настоящий пользователь |
| **Семинар** | 40 минут, кто-то из команды объясняет остальным одну тему на нашем коде. Ведёт не обязательно лид |

### Технические слова (подробнее — в `docs/TZ.md` §1)

| Слово | Что значит одной строкой |
|---|---|
| **Событие (event)** | факт, который нам прислали и который надо разослать |
| **Сообщение (message)** | «доставить событие X получателю Y». Одно событие → много сообщений |
| **Фан-аут** | размножение события в сообщения по числу подписчиков |
| **Очередь** | таблица `messages` в базе. Строки со статусом `pending` и есть очередь |
| **Воркер** | горутина, которая в цикле берёт из очереди и доставляет |
| **Claim / Ack / Nack** | «захватить задачу» / «доставил успешно» / «не доставил, повторим позже» |
| **Лиз (lease)** | временная бронь на сообщение (30 сек). Воркер упал — бронь протухла — сообщение вернулось |
| **Reaper («жнец»)** | фоновая горутина, возвращающая в очередь сообщения с протухшей бронью |
| **SKIP LOCKED** | приписка в SQL, благодаря которой воркеры не дерутся за одни и те же строки |
| **Backoff** | растущая пауза между повторами: 5 сек, 10, 20, 40… |
| **Jitter** | случайная добавка к паузе, чтобы повторы не сошлись в один залп |
| **DLQ** | «склад невостребованного»: сообщения, не доставленные после всех попыток |
| **Replay** | кнопка «попробовать доставить ещё раз» вручную |
| **Circuit breaker** | предохранитель: перестаём слать на получателя, который стабильно падает |
| **HMAC-подпись** | отпечаток тела запроса на общем секрете — получатель проверяет, что запрос от нас |
| **Идемпотентность** | повтор операции не меняет результат (защита от дублей) |
| **Graceful shutdown** | корректное выключение: доделали текущее, потом вышли |
| **Dogfooding** | «есть свой корм»: пользоваться собственным продуктом. У нас — прогнать вебхуки своего же репозитория через свой сервис |
| **Keyset-пагинация** | листание страниц через «покажи то, что старше вот этой записи» вместо `OFFSET` |
| **Snippet (снипет)** | обрезанный кусочек тела ответа, который сохраняем в журнал (1 КБ) |

**Как читать таблицы задач ниже.** Одна строка = одна-три задачи на доске. Колонки: что сделать · кто делает · уровень сложности. У каждой задачи есть пояснение простыми словами — читай его, если название непонятно.

# 5. Фаза 0 — Фундамент (неделя 1)

## Что происходит простыми словами

Кода почти нет. Эта неделя про то, чтобы:
1. проект **запускался** у всех на компьютере одной командой;
2. робот на GitHub **проверял** каждый PR;
3. на бумаге был согласован **дизайн очереди** — до того, как написана первая строчка кода доставки.

🧠 **Почему дизайн очереди пишем раньше кода.** Очередь — сердце проекта, от неё зависят все остальные модули. Если спроектировать её неправильно и понять это на пятой неделе — переписывать придётся половину. Полтора часа на бумаге экономят неделю кода.

# 6. Фаза 1 — Первая доставка (недели 2–3)

## Что происходит простыми словами

Самая важная фаза. К её концу событие должно **пройти весь путь**: пришло по HTTP → сохранилось в базу → размножилось по подписчикам → воркер его забрал → отправил получателю с подписью → получатель принял.

Пока без повторов при ошибках — только «счастливый путь». Ретраи будут в фазе 2.

🧠 После этой вехи проект считается состоявшимся. Дальше — наращивание.

## Задачи

Работа идёт **двумя параллельными дорожками**, они не мешают друг другу.

### 🟦 Дорожка «ядро» (лид и стажёр)

| Задача | Что это значит простыми словами | Кто | Ур. |
|---|---|---|---|
| `queue`: захват задач | SQL-запрос, которым воркер забирает себе пачку сообщений так, чтобы другому воркеру они не достались. Плюс «отчёты» об успехе и неудаче | Лид | L3 |
| `queue`: тест на два воркера | проверка, что при двух одновременных воркерах одно сообщение не уходит обоим | Лид | L3 |
| `worker`: пул воркеров | несколько горутин, каждая в цикле «взял → доставил → отчитался». Плюс корректная остановка по сигналу | Лид | L3 |
| `ingest`: приём события | HTTP-хендлер: принять JSON, проверить, сохранить в базу, ответить `202` | Ст | L2 |
| `ingest`: фан-аут | найти всех подписчиков этого типа события и создать по сообщению на каждого — в одной транзакции с самим событием | Ст | L2 |
| `delivery`: отправка получателю | HTTP-клиент: собрать запрос с подписью, отправить с таймаутом 10 с, аккуратно прочитать ответ (не больше 1 МБ) | Ст | L2 |

### 🟩 Дорожка «чистые модули» (новички — **можно начинать сразу, ничего не ждём**)

| Задача | Что это значит простыми словами | Кто | Ур. |
|---|---|---|---|
| `signing` | функция, которая ставит на запрос «печать» (HMAC-подпись), и вторая, которая печать проверяет. В ТЗ §6.2 есть готовые тестовые векторы — сверяешь по ним | A | L2 |
| `backoff` | функция «через сколько повторить попытку номер N»: 5 сек, 10, 20, 40… плюс случайный разброс | A | L1 |
| `matcher` | функция «подходит ли событие `order.created` под подписку `order.*`». Чистая логика на строках | A | L1 |
| HTTP-каркас | обвязка сервера: логирование запросов, единый формат ошибок, защита от падения при панике | B | L1 |
| Admin API | эндпоинты для настройки: создать приложение, добавить получателя, добавить подписку | B | L2 |
| `cmd/sink` | маленький «фальшивый получатель» для тестов: умеет по команде отвечать 200, 500 или зависать. **Пользоваться им будут все** | C | L1 |
| `attempts` | запись в журнал каждой попытки доставки + эндпоинт «покажи историю по сообщению» | C | L2 |
| Семинар №1: «слои, интерфейсы, ошибки» | 40 минут, ведёт стажёр, на нашем коде | Ст | — |

## 🟩 Если ты новичок — что делать в эту фазу

Твои задачи (`signing`, `backoff`, `matcher`, `sink`, HTTP-каркас) — **чистые функции**. Им не нужны ни база, ни docker, ни очередь. Даже если инфраструктура ещё не готова, ты пишешь код и гоняешь тесты командой:

```bash

```

Порядок работы над своей задачей:
1. Открой `docs/TZ.md`, найди раздел про **свой** модуль (§6.2 signing, §6.3 backoff, §6.4 matcher, §6.12 sink).
2. Там есть готовая сигнатура функции, алгоритм по шагам и список обязательных тестов.
3. **Сначала напиши тест, потом код.** Для этих модулей так удобнее: в ТЗ уже перечислено, что проверять.
4. Прогони `make test` и `make lint` до открытия PR.

⚠️ Не пиши `time.Now()` внутри своих функций — время передавай параметром. Почему — в ТЗ §6.10. Это правило проекта, на ревью проверяют.

# 7. Фаза 2 — Надёжность (недели 4–5)

## Что происходит простыми словами

Учим сервис **переживать сбои**. До этого мы умели доставить, когда всё хорошо. Теперь:

- получатель ответил ошибкой → повторяем через 5 секунд, потом через 10, 20, 40…;
- после 8 неудач сдаёмся и кладём сообщение «на склад невостребованного» (DLQ), откуда человек может запустить повтор кнопкой;
- воркер упал посреди доставки → сообщение не потерялось, его подхватил другой;
- всё это видно человеку в дашборде.

## Задачи

| Задача | Что это значит простыми словами | Кто | Ур. |
|---|---|---|---|
| Перенос неудачных сообщений | при ошибке доставки: посчитать паузу через `backoff` и вернуть сообщение в очередь «на потом» | Лид | L3 |
| **Reaper** («жнец») | фоновая горутина: раз в 15 секунд ищет сообщения, которые кто-то захватил и не вернул (воркер умер), и возвращает их в очередь | Лид (+пара со Ст) | L3 |
| Тест на падение воркера | буквально убиваем контейнер воркера в момент доставки и проверяем, что сообщение доехало другим | Лид + Ст | L3 |
| DLQ | после исчерпания попыток сообщение получает статус `dead` + эндпоинт «покажи все мёртвые» | Ст | L1 |
| Replay | кнопка «повторить»: создаёт новое сообщение со ссылкой на исходное | Ст + B | L2 |
| Дашборд: лента и карточка | страница со списком событий и страница одного сообщения со всей историей попыток | B | L2 |
| Дашборд: проверка подписи | страничка, куда можно вставить тело + секрет + подпись и получить «валидна / нет» | B + A | L1 |
| Пагинация | листание длинных списков без тормозов (через курсор, а не `OFFSET` — почему, в ТЗ §6.13) | C | L2 |
| Обрезка тела ответа | сохраняем в журнал только первый килобайт ответа, аккуратно, не разрезая символы пополам. И вычищаем секреты из заголовков | C | L1 |
| Нагрузочный прогон | проверить, что 500 сообщений в минуту на трёх воркерах не копятся в очереди | Лид | L3 |
| Семинар №2: «горутины, каналы, context» | 40 минут, ведёт лид, на коде воркера. **Готовит новичков к их первым L3-задачам** | Лид | — |

## 🟩 Если ты новичок — что делать в эту фазу

Ты уже закрыл 2–3 задачи и понимаешь процесс. Теперь:
1. Обязательно приди на **семинар №2** — он готовит к конкурентности.
2. Возьми задачу уровня L2 (пагинация, обрезка ответа, страница дашборда).
3. Скажи лиду, что хочешь L3 в фазе 3, и договорись о паре. Не жди, пока предложат.

## ✅ Фаза закрыта, когда

Можно показать такой фокус на демо:
1. Выключаем тестовый приёмник (`sink`).
2. Шлём события — видим в дашборде, как сообщения копятся и повторяются со всё большими интервалами.
3. Включаем приёмник обратно — всё доезжает само.
4. Отдельное сообщение доводим до исчерпания попыток — оно в DLQ.
5. Жмём **Replay** — оно доставляется.

## 📋 На доске в эту фазу

- **Начнут появляться флакующие тесты** (падают через раз). Это ожидаемо: мы работаем со временем и многопоточностью. Каждый такой тест = отдельная задача с меткой `flaky` и ссылкой на упавший прогон.
- ⚠️ **Правило:** перезапускать CI «пока не позеленеет» — запрещено. Почти каждый флак здесь вскрывает настоящую гонку в коде. Заведи вью `⚠️ Flaky` и смотри на планировании: если там больше трёх задач — неделя уходит на починку тестов, новые фичи не берём.
- **Контрольная точка (конец недели 5):** если в команде осталось ≤ 3 человека — режем circuit breaker, ограничение частоты и Telegram-демо. Ядро остаётся: приём + очередь + воркеры + ретраи + DLQ + дашборд. Это уже полноценный проект. Решение записывается в отдельную задачу, не устно.

# 8. Фаза 3 — Масштаб и живое демо (недели 6–7)

## Что происходит простыми словами

1. Запускаем **три воркера одновременно** (три контейнера) и доказываем, что они не мешают друг другу.
2. Добавляем предохранитель: если получатель стабильно лежит, перестаём его долбить.
3. Делаем **живое демо**: подключаем вебхуки нашего собственного GitHub-репозитория к нашему же сервису, а получателем ставим Telegram-бота. Пуш коммита → сообщение в чате, прошедшее через нашу очередь.
4. Вешаем метрики и графики.

## Задачи

| Задача | Что это значит простыми словами | Кто | Ур. |
|---|---|---|---|
| Три воркера в docker-compose + тест | запустить три контейнера воркеров и доказать тестом: 200 сообщений, каждое доставлено ровно один раз | Лид | L3 |
| Ограничение частоты | не слать одному получателю больше N запросов в секунду. Сложность: воркеров несколько, а лимит общий | Лид + A | L3 |
| **Circuit breaker** | предохранитель: 10 неудач подряд → перестаём слать на 5 минут → пробуем одно сообщение → решаем, чинить или ждать дальше | Ст | L3 |
| Воркер-уборщик | фоновая горутина, удаляющая старые события пачками, чтобы база не пухла. **Первая L3-задача для A** | A (+пара с Лидом) | L3 |
| Приём вебхуков GitHub | отдельный эндпоинт, понимающий формат GitHub: проверить их подпись, взять тип события из заголовка | A + Ст | L2 |
| **Dogfooding** | подключить вебхук своего репозитория к своему сервису (как — в блоке «На доске» ниже) | C (+B) | L2 |
| Telegram-подписчик | маленький сервис, который принимает наш вебхук и пишет в чат. **Первая L3-задача для C** | C | L3 |
| Метрики и графики | счётчики «сколько ждёт», «сколько умерло», «возраст самого старого ждущего» + дашборд Grafana | C + B | L2 |
| Дашборд: статусы получателей | показать, у кого предохранитель разомкнут. **Первая L3-задача для B** | B (+пара со Ст) | L3 |
| Корректная остановка под нагрузкой | по сигналу выключения: доделать текущие доставки, остальное честно вернуть в очередь | Лид | L3 |

## 🟩 Если ты новичок — это твоя главная фаза

Здесь **каждый** из A, B, C берёт свою первую задачу с конкурентностью (L3). Как к ней подойти:

1. **Обязательно в паре.** Договорись с лидом или стажёром на созвон с шарингом экрана, 1–1,5 часа. Не «я сам разберусь» — здесь это не работает.
2. Перечитай `docs/TZ.md` §6.10 (про `Clock`) и раздел про свой модуль.
3. Правила, которые спасут тебя от 90% проблем:
   - у каждой горутины должен быть понятный **способ её остановить** (через `context`);
   - никакого `time.Sleep` в тестах — только фейковые часы;
   - гоняй `go test -race` — он ловит гонки, которые глазами не видны.
4. Берёшь **одну** L3 за спринт, не две. Они выматывают.

## ✅ Фаза закрыта, когда

Делаешь пуш в репозиторий → через несколько секунд в Telegram падает сообщение, прошедшее через вашу очередь. В Grafana видно всплеск. И — главный трюк — прямо в этот момент убиваешь один из трёх воркеров, и **ничего не ломается**.

Вот это и снимается на гифку для README.

## 📋 На доске в эту фазу

- **Как настроить dogfooding:** Settings репозитория → **Webhooks** → Add webhook. Payload URL = публичный адрес вашего приёма (`https://.../ingest/github/{app_id}`), Content type = `application/json`, Secret = секрет приложения, события = `push` и `pull_request`.
- **Где отлаживать:** в настройках вебхука есть вкладка **Recent Deliveries** — там видно каждый отправленный GitHub'ом запрос, ваш ответ на него, и есть кнопка **Redeliver** (отправить заново). Это бесплатный отладчик, о котором мало кто знает.
- **Нет публичного адреса?** Не надо ради этого поднимать сервер: используй туннель (`ngrok` или `cloudflared`), он даёт временный публичный URL на твой localhost.
- ⚠️ **Токен Telegram-бота никогда не коммить.** Только в переменные окружения. GitHub его поймает и заблокирует пуш, но лучше не проверять.

# 11. Приложение для тимлида: как роадмап живёт в GitHub

> 🟨 Этот раздел — только для тимлида. Остальным читать не обязательно.

## 11.1 Соответствие: роадмап ↔ GitHub

| В этом файле | В GitHub | Кто заводит |
|---|---|---|
| **Фаза** (0–4) | **Milestone** (Issues → Milestones) | лид, один раз в фазе 0 |
| **Неделя** (S1…S8) | **Iteration** в поле `Sprint` на доске | лид, генерируются разом |
| **Строка таблицы** | 1–3 **Issue** с полями Level / Size / Module | лид на груминге |
| **Exit-критерий** | описание Milestone + якорный issue `Exit фазы N` с чек-листом | лид |

⚠️ **Не путать Milestone и Sprint.** Milestone отвечает на вопрос «какой кусок продукта делаем» (фаза 2 — ретраи и DLQ), длится 1–2 недели, закрывается по Exit-критерию. Sprint отвечает «какая сейчас неделя», нужен для оценки загрузки. У одной задачи есть **и то, и другое**.

## 11.2 Разовая настройка (≈40 минут, в фазе 0)

**1. Пять Milestone.** Issues → Milestones → New milestone. Название, дата и Exit-критерий — прямо в описание:

| Milestone | Дедлайн | Описание (копировать из соответствующего «Фаза закрыта, когда») |
|---|---|---|
| `Фаза 0 — Фундамент` | конец недели 1 | CI зелёный; `make up && make test` у всех; ADR-0003 смержен; ≥6 good-first-issue в Ready |
| `Фаза 1 — Первая доставка` | конец недели 3 | curl → sink получил подписанный запрос; тест «2 воркера, 100 сообщений» зелёный |
| `Фаза 2 — Надёжность` | конец недели 5 | выключили sink → ретраи; включили → доехало; DLQ реплеится кнопкой |
| `Фаза 3 — Масштаб и демо` | конец недели 7 | пуш → Telegram через нашу очередь; убийство воркера ничего не ломает |
| `Фаза 4 — Релиз` | конец недели 8 | тег v1.0.0; каждый объяснил свои строки ТЗ §13 |

**2. Итерации.** Доска → Settings → поле `Sprint` (тип **Iteration**) → длительность 1 неделя, старт понедельник → сгенерировать 8, переименовать в S1…S8.

**3. Вью «Роадмап».** На доске `+ New view` → **Roadmap** → в настройках: Date fields = `Sprint`, Group by = `Milestone`, Markers = Milestones. Получится живой таймлайн — его и показываешь на планировании вместо этого файла.

**4. Вью «Загрузка».** `+ New view` → **Table** → Group by = `Assignee`, фильтр `sprint:@current`. Сразу видно, кто перегружен, а кто простаивает.

**5. Пять якорных issue** `Exit фазы N` — с чек-листом критериев внутри, привязанные к своим Milestone. Закрываешь такой issue только когда фаза реально сдана.

| Когда | Что делаешь | Сколько |
|---|---|---|
| чт–вс | **Груминг:** взять 5–8 верхних задач из `Backlog`, дописать критерии приёмки, проставить `Level`/`Size`/`Module`/`Milestone`, перевести в `Ready` | 15 мин |
| пн | **Планирование:** открыть вью `Спринт`, проставить взятым задачам `Sprint = текущая`, каждый ставит себя в Assignee | 30–45 мин |

## 11.4 Как видеть прогресс без созвонов

- **Страница Milestone** сама рисует прогресс-бар «12 из 18 закрыто» — ответ на «успеваем ли по фазе».
- **Insights** на доске даёт график по итерациям — velocity считается автоматически.
- **Вью «Загрузка»** показывает, у кого 4 задачи `In Progress` (нарушение правила «максимум 2») и кто простаивает.

## 11.5 Правила ведения самого роадмапа

- Файл живёт в репозитории, правки — только через PR, как код.
- Сдвиг вехи больше чем на неделю = обязательный пересмотр объёма работ на ретро, а не молчаливый перенос.
- Задачи новичков всегда должны быть в `Ready` с запасом. Пустая колонка `Ready` в понедельник — главная управленческая ошибка в этом проекте.
