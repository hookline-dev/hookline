# STARTER TASKS — задачи для новичков Hookline (L0 → ранний L1)

Безопасная песочница: ничего из этого не трогает `queue/` и `worker/`, ничего не блокирует ядро, но всё реально нужно проекту. Лид заводит их как issue с меткой `good-first-issue` и полем Level = L0/L1.

Приятная особенность Hookline: половина по-настоящему ценных модулей — **чистые функции без I/O**. Их пишут новички с полным TDD, и именно про них потом рассказывают на собеседовании.

## Блок 3. Первый код на Go — чистые функции, ценные для резюме (недели 2–3)

Это не «песочница ради песочницы»: три задачи ниже — реальные модули из ТЗ.

**T-09. `backoff`: калькулятор задержки** — M, L1
Чистая функция `Next(attempt int, base, cap time.Duration, rnd *rand.Rand) time.Duration`: экспонента `base*2^attempt`, ограничение `cap`, **full jitter** (`rand(0, delay)`).
Приёмка: table-driven тесты с фиксированным seed rand: attempt=0..10, проверка что не превышает cap, что при jitter значение в диапазоне, что attempt=0 даёт минимальную задержку. 100% покрытие. В комментарии к функции — 2 строки про thundering herd (потом расскажешь у доски).

**T-10. `matcher`: матчинг типов событий** — M, L1
`Match(eventType string, pattern string) bool`: точное совпадение, `order.*`, `*`, регистр, пустые значения. Плюс `MatchAny(eventType, patterns []string)`.
Приёмка: ≥15 кейсов включая каверзные (`order.` vs `order.*`, `*.created`), 100% покрытие.

**T-11. `signing`: верификация подписи** — M, L1→L2
Функция `Verify(payload []byte, header string, secret string, now time.Time, tolerance time.Duration) error`: разбор заголовка `v1=...`, пересчёт HMAC-SHA256, **`hmac.Equal`** (не `==`!), проверка возраста timestamp.
Приёмка: тесты на валидную подпись, битую подпись, чужой секрет, старый timestamp, мусор в заголовке. 100% покрытие. Подписывающую часть делает A — согласуйте формат заранее.

**T-12. `cmd/sink` — управляемый приёмник** — M, L1
Маленький HTTP-сервис: `POST /hook?status=500&delay=3s` → отвечает заданным кодом с задержкой; всё полученное пишет в лог и в память; `GET /received` — что пришло.
Приёмка: используется в интеграционных тестах и в `make demo`. Это самый полезный инструмент проекта — им будут пользоваться все.

**T-13. Godoc-комментарии пакета `domain`** — S, L0→L1
Каждому экспортируемому типу/ошибке — комментарий по конвенции Go.

**T-14. `/readyz`** — M, L1
Пинг БД + (для воркер-режима) «есть ли живые воркеры»; JSON со статусами компонентов; юнит-тест с мок-пингером.

**T-15. Обрезка и редактирование данных в `attempts`** — M, L1
Тело ответа обрезать до 1 КБ (аккуратно по рунам, не резать UTF-8 посередине!); из сохраняемых заголовков вычищать значения `Authorization`/`X-Hookline-Signature`.
Приёмка: тесты на многобайтные символы и на редактирование секретов.

## Правила для лида

1. В issue уровней L0–L1 обязательны подсказки: файл, похожий код в репо, ссылка на нужный раздел документации Go.
2. Задачи T-09…T-11 — не «детские»: это модули из ТЗ. Проговори это вслух на планировании, чтобы новички понимали, что пишут настоящий продакшн-код, а не упражнение.
3. Ревью стартовых задач ≤ 24 ч, тоном «предлагаю», обязательно отмечай хорошее.
4. Выпуск новичка в общий бэклог: 2 задачи блоков 1–2 + 2 задачи блока 3 → объявляется на пятничном демо.
5. Стартовые задачи не бывают срочными: неделя на T-задачу — норма, давить нельзя.
